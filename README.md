# ACPs-TrainingCamp-冀婉禾

## 项目简介

本项目展示了如何使用 **Server-Sent Events (SSE)** 替代原demo中的轮询机制，优化前端与后端的实时通信。这是对 ACPs-Demo-Project项目中轮询逻辑的替代方案。

本项目还包含了贪心算法的实现，用于解决活动选择问题。
## 中断机制实现

## 核心思路

### 问题分析
**注意到原项目ACPs-Demo-Project中Web App使用轮询机制，**
- **轮询**：前端定时向后端发送请求，即使状态没有变化也会发送，导致：
  - 实时性差（固定间隔）
  - 服务器负载高（不必要的请求）
  - 网络流量大（频繁的请求/响应）

### 中断方案
- **SSE 替代方案**：后端主动向前端推送状态更新，只有在状态变化时才发送数据，优势：
  - **实时性**：服务器有新数据时立即推送
  - **效率高**：减少不必要的网络请求
  - **服务器负载低**：只在有数据时发送
  - **客户端资源消耗少**：不需要持续的定时器

## 核心代码说明

### 前端实现 (`sse_demo/frontend_sse.js`)
- 使用 `EventSource` API 建立 SSE 连接
- 监听 `message` 事件接收服务器推送的状态
- 监听 `complete` 事件处理任务完成
- 监听 `error` 事件处理连接错误

### 后端实现 (`sse_demo/backend_sse.py`)
- 使用 FastAPI 实现 SSE 端点
- 通过 `StreamingResponse` 持续推送事件
- 模拟任务处理过程，实时更新状态
- 任务完成时发送 `complete` 事件

## 贪心算法实现
 
### 实现文件 (`greedy_algorithm/activity_selection.py`)
- **算法原理**：使用贪心策略，按活动结束时间排序，选择不重叠的活动
- **核心步骤**：
  1. 按结束时间对活动进行排序
  2. 选择第一个活动（结束时间最早的）
  3. 遍历剩余活动，选择与已选活动不重叠的活动
- **时间复杂度**：O(n log n)，主要用于排序
- **空间复杂度**：O(n)，用于存储排序后的活动列表
 
### 示例测试
算法包含了一个示例测试，演示如何选择最多的不重叠活动。
 
### 应用场景
- 调度问题：如会议安排、任务调度等
- 资源分配：如共享资源的使用安排
- 时间管理：如个人时间规划
